# WritersLogic — Product and Technology Overview

WritersLogic is a cryptographic authorship witnessing system that captures behavioral evidence during the act of writing and packages it into tamper-evident, cryptographically signed evidence packets. Rather than attempting to detect whether content was written by a human or generated by AI after the fact, WritersLogic proves authorship by continuously witnessing the creative process as it happens.

The macOS application, branded **WritersProof**, runs as a lightweight menu bar sentinel that monitors supported writing applications, captures behavioral signals at the keystroke level, and produces court-ready evidence of human authorship. The system implements the **draft-condrey-rats-pop** IETF protocol specification (Proof-of-Process) and is covered by USPTO Application No. 19/460,364.

## Why Attestation, Not Detection

AI content detection and cryptographic process attestation solve fundamentally different problems, and the distinction matters.

### The Detection Problem

AI detectors analyze finished text and attempt to classify it as human-written or machine-generated. This approach has structural limitations that cannot be overcome with better models:

- **No ground truth.** A detector examines the output in isolation. It has no knowledge of how the text was produced, only statistical properties of the result. Two identical paragraphs, one typed by a human and one generated by GPT, are indistinguishable at the output level if the human happens to write in a style the model also produces.
- **Adversarial fragility.** Simple paraphrasing, light editing, or mixing human and AI text defeats most detectors. The arms race between generators and detectors favors the generator because the detector must work from less information.
- **False positives harm real people.** Non-native English speakers, formulaic writing styles, and technical prose routinely trigger false AI detection. A system that accuses real authors of fraud is worse than no system at all.
- **No chain of custody.** Even a correct detection result carries no cryptographic proof. It is an opinion, not evidence. It cannot be independently verified, cannot be presented in court, and degrades as AI models improve.

### The Attestation Advantage

WritersLogic inverts the problem. Instead of asking "does this text look human?" after the fact, it asks "was a human observed producing this text?" during the act of writing.

- **Process, not product.** The system captures behavioral evidence (keystroke timing, revision patterns, cognitive pauses, editing topology) as the text is being written. This evidence is bound to the specific document through cryptographic hash chains.
- **Unforgeable physics.** Human typing exhibits fractal timing patterns (Hurst exponent ~0.7), 1/f pink noise spectral characteristics, and error-correction reflexes that are computationally expensive to simulate. The system measures these biological invariants continuously.
- **Cryptographic proof chain.** Every checkpoint is hash-chained to its predecessor, signed with a ratcheting key hierarchy, and optionally bound to hardware (Secure Enclave / TPM). Tampering with any link invalidates all subsequent links.
- **Independent verifiability.** Evidence packets are self-contained CBOR/COSE documents that any third party can verify without trusting WritersLogic. The math either checks out or it does not.
- **Graceful degradation.** When full behavioral evidence is not available, the system still provides signed declarations with clear tier labeling. A Declared attestation is transparently weaker than a Verified one, and the system says so.

Detection answers "what does this look like?" Attestation answers "what actually happened?" The first is a guess; the second is a proof.

## Application Overview

### Menu Bar Sentinel

WritersProof runs as a macOS menu bar application. Left-clicking the icon opens a popover dashboard; right-clicking shows recent tracked files and quick navigation.

The menu bar icon reflects the current state:

| State | Meaning |
|-------|---------|
| Not Initialized | Engine has not completed setup |
| Ready | Initialized, no active document |
| Active | Sentinel is monitoring a document |
| Detached | Popover has been torn off into a floating window |
| Error | Engine fault |

The icon flashes briefly when a checkpoint is created.

### Onboarding

First launch presents a four-phase onboarding flow:

1. **Privacy Declaration** — explains what the system captures and, critically, what it does not (no content, no screenshots, no network transmission of text).
2. **Welcome** — introduces the core value proposition: cryptographic proof of authorship.
3. **Setup** — initializes the engine, requests Accessibility and Input Monitoring permissions, and calibrates the Sequential Work Function (SWF) baseline for the user's hardware.
4. **Live Demo** — shows the real-time keystroke feed so the user can see exactly what data is captured.

### Dashboard

The main screen after onboarding displays:

- **Evidence Timeline** — a progress bar showing keystroke accumulation across five thresholds (50, 200, 500, 2,000, 5,000 keystrokes), indicating evidence strength.
- **Active Document Stats** — live keystroke count with pulse animation, evidence strength percentage derived from forensic scoring, and time since last checkpoint.
- **Quick Actions** — four buttons for Checkpoint, Snapshot, Export, and History.
- **Paste Detection Banner** — when a large paste is detected, the popover pins open and presents a mandatory acknowledgment flow. The user must either describe the paste source and save a checkpoint, or explicitly dismiss with a warning. This ensures paste events are never silently absorbed into the evidence chain.
- **System Status** — SWF calibration state, TPM/Secure Enclave availability, and total checkpoint count.

### Checkpoint Form

Creating a checkpoint requires:

- **File selection** via picker, drag-and-drop, or pre-filled from the right-click context menu.
- **Optional note** (up to 500 characters, control characters and Unicode format characters stripped).
- **Mandatory authorship declaration** — one of three options:
  - "No AI Tools Used" (tag: `[tool:none]`)
  - "AI-Assisted" with specific tool selection: ChatGPT, Claude, GitHub Copilot, Gemini, or Other (tag: `[tool:ai:ToolName]`)
  - "Reference Material Only" (tag: `[tool:reference]`)

The tool declaration is cryptographically embedded in the checkpoint. No checkpoint can be committed without a declaration.

### History

A split-pane view with searchable file list (300ms debounce) on the left and checkpoint detail on the right. Files can be filtered by verification status (All / Verified / Pending / Failed) and sorted five ways. Right-click context menus on files offer Export Evidence, Verify Evidence, Copy File Path, and Show in Finder.

### Batch Verification

Accepts up to 200 evidence files (`.cpoe`, `.cwar`, `.json`) via picker or drag-and-drop. Runs four concurrent verifications. Requires biometric authentication before starting. Each item shows filename, status, and expandable verdict details.

### Export

Three export formats, all requiring biometric authentication and a pre-export tamper check:

1. **PDF Report** — formatted authorship report saved alongside the source document.
2. **Proof Block** — PGP-style ASCII-armored text block copied to clipboard:
   ```
   -----BEGIN WRITERSPROOF PROOF-----
   ...
   -----END WRITERSPROOF PROOF-----
   ```
3. **Evidence Package** — full `.cpoe` archive with all checkpoint data, behavioral evidence, and cryptographic signatures. Sets the source file read-only and anchors to WritersProof infrastructure.

Export tiers control evidence depth: Core (1), Enhanced (2), Maximum (3).

### Voice Fingerprint

Opt-in writing style analysis that builds an author profile from typing rhythm patterns. Displays live metrics (WPM, inter-key interval, think time, correction rate) and progress toward a 500-sample baseline. The fingerprint can be exported as JSON. All data stays on-device; only statistical patterns are stored, never raw text.

### Data Transparency

A real-time feed showing exactly what the sentinel captures:

- Scrolling event log with millisecond timestamps, keyboard zone indicators, and color-coded intervals (red for suspiciously fast, green for normal human range, gray for pauses).
- Keyboard zone diagram showing the five capture zones (Thumb/Modifier, Left Home Row, Right Home Row, Left Reach, Right Reach).
- Explicit list of what is NOT captured.
- Storage explanation (local encrypted SQLite).

### Settings

Five tabs:

- **Account** — authentication, witnessing statistics, hardware security status (Secure Enclave Active vs. Software Key Store), cloud sync controls, BIP-39 recovery phrase display, device attestation history.
- **General** — launch at login, global hotkey, popover size, notification preferences.
- **Monitoring** — checkpoint interval, minimum keystroke threshold, paste detection sensitivity, SWF recalibration.
- **Security & Privacy** — data directory integrity checks with hash verification, repair capability, and exportable integrity reports.
- **Storage** — local database statistics, history management, snapshot controls.

### Right-Click Context Attestation

WritersProof registers as a macOS Services provider, adding attestation options to the system-wide right-click (Services) menu in any application. When a user selects text and invokes the service, the system evaluates the available evidence and produces an attestation at the appropriate tier:

## How Evidence Is Captured

### Real-Time Behavioral Monitoring

The sentinel daemon runs a single-threaded async event loop consuming five concurrent streams:

1. **Keystroke Events** — captured via macOS `CGEventTap` and IOKit HID Manager (dual-layer validation). Every keyDown produces a `SimpleJitterSample` containing: nanosecond timestamp, interval since previous keystroke, keyboard zone, dwell time (key-down duration), and flight time (previous keyUp to current keyDown). Each keystroke feeds the incremental jitter hash chain and the behavioral entropy accumulator.

2. **Focus Events** — 100ms polling of the frontmost application and document. Creates and ends document sessions as focus changes. Tracks application bundle identifiers, document paths, and window titles.

3. **Mouse Events** — micro-movements during typing captured at ~20Hz. A steganographic engine embeds covert HMAC-SHA256 session signatures into mouse event timing (sub-perceptual delays of 500-2000 microseconds), creating a verifiable secondary evidence channel.

4. **Clipboard Events** — paste operations are detected through three independent signals: keystroke silence (typing stops then large content appears), content hash discontinuity (document hash jumps without proportional keystrokes), and application transition (focus switch to/from known AI tools). Two of three signals must agree before flagging a paste event.

5. **Checkpoint Timer** — at configurable intervals (default 30-60 seconds), the system commits a checkpoint for all active sessions with new keystrokes. Each checkpoint includes a server challenge nonce with 30-second TTL, proving the checkpoint hash was unknown to the server at nonce issuance time.

### Behavioral Signals

The system captures and analyzes:

- **Inter-Keystroke Intervals (IKI)** — timing between consecutive keystrokes, measured in microseconds. Mean, standard deviation, coefficient of variation, and full distribution histogram.
- **Keyboard Zone Transitions** — movement patterns across five keyboard zones, encoded as zone-pair transitions. Same-finger, same-hand, and alternating-hand transition histograms.
- **Typing Bursts and Pauses** — burst detection (rapid sequences) and pause detection (>2 seconds), with counts and distributions.
- **Edit Topology** — spatial map of where in the document edits occur, encoded as `{start_pct, end_pct, delta_sign, byte_count}` regions.
- **Revision Patterns** — insertion/deletion ratios, deletion clustering coefficients, cursor movement patterns, undo/redo counts.
- **Cognitive Signals** — sentence-boundary delays, paragraph transition pauses, thinking pause frequency, IKI modality analysis.
- **Hurst Exponent** — fractal analysis of keystroke timing. Human typing exhibits long-range dependence (H ~0.7); scripted input shows H ~0.5 (white noise) or H ~1.0 (deterministic).
- **1/f Pink Noise Analysis** — power spectral density of IKI sequences. Human motor control produces 1/f noise with spectral slope alpha in [0.8, 1.2]; synthetic input does not.
- **Error Topology** — spatial clustering of backspace/delete sequences relative to the error location. Human typos cluster near adjacent keys with characteristic correction latencies.
- **Perplexity Scoring** — character-level n-gram model trained on the author's own writing history. Stylistically anomalous text (potential AI insertion) produces elevated perplexity.

### Three-Layer Persistence

Evidence flows through three persistence layers:

1. **RAM Buffer** — high-frequency capture (every keystroke), flushed every 100ms or 1,000 entries.
2. **Write-Ahead Log (WAL)** — append-only, fsync'd, HMAC integrity on each entry. Survives crashes.
3. **MMR Checkpoint** — permanent record in SQLite with Merkle Mountain Range proofs. The MMR root is committed into each checkpoint hash, so deleting any checkpoint invalidates all subsequent hashes.

### Composite Authorship Score

Evidence strength is computed as a weighted composite:

```
ProcessScore = 0.3 × Residency + 0.3 × Sequence + 0.4 × Behavioral
```

- **Residency (30%)** — total duration and continuity of the editing session.
- **Sequence (30%)** — hash-chain integrity and edit topology entropy.
- **Behavioral (40%)** — keystroke cadence analysis, coefficient of variation, burst detection, and biological invariant checks.

| Score | Verdict |
|-------|---------|
| >= 0.90 | Manual Composition Consistent (Verified Human) |
| 0.70-0.90 | Manual Composition Likely |
| 0.40-0.70 | Inconclusive |
| < 0.40 | Automated/Retyped Content Likely |

## Evidence Formats

### .cpoe — Evidence Packet

The primary evidence format. Wire encoding is CBOR (RFC 8949) with a custom CBOR tag `1129336645` (ASCII "CPoE"). MIME type: `application/vnd.writersproof.cpoe+cbor`. File extension: `.cpoe`.

Each packet is a COSE_Sign1 signed envelope containing:

| CBOR Key | Field | Description |
|----------|-------|-------------|
| 1 | version | Protocol version (currently 1) |
| 2 | profile-uri | EAT profile identifier |
| 3 | packet-id | 16-byte UUID |
| 4 | created | Epoch milliseconds |
| 5 | document-ref | Document title, path, final hash, final size |
| 6 | checkpoints | Array of 3 to 10,000 checkpoint records |
| 7 | attestation-tier | T1-T4 hardware assurance level |
| 8 | limitations | Array of strings describing what the evidence cannot prove |
| 9 | profile-declaration | Author's process declaration |
| 10 | presence-challenges | Active probe results |
| 11 | channel-binding | Secure channel binding data |
| 12 | signing-public-key | 32-byte Ed25519 public key |
| 13 | content-tier | Core (1), Enhanced (2), Maximum (3) |
| 18 | physical-liveness | Physical context evidence |
| 19 | baseline-verification | Zero-trust behavioral baseline check |
| 20 | author-did | Decentralized identifier (must start with "did:") |
| 21 | document-content | Embedded document content (up to 100MB) |
| 22 | document-filename | Original filename |
| 23 | project-file-refs | Related files in the same project |

**Signing:** The content hash is computed as `SHA-256("cpoe-packet-content-v3" || CBOR(packet_without_signature_fields))`. If a verifier nonce is present, the signing payload becomes `SHA-256("cpoe-nonce-binding-v1" || content_hash || nonce)`, proving the packet was constructed after the nonce was issued.

#### Checkpoint Structure

Each checkpoint in the chain carries:

| CBOR Key | Field | Description |
|----------|-------|-------------|
| 1 | sequence | Monotonic ordinal from 0 |
| 2 | checkpoint-id | 16-byte UUID |
| 3 | timestamp | Epoch milliseconds |
| 4 | content-hash | SHA-256 of document content at this point |
| 5 | char-count | Document character count |
| 6 | edit-delta | Characters added, deleted, operation count, cursor trajectory histogram, revision depth histogram, pause duration histogram |
| 7 | prev-hash | Hash of previous checkpoint (genesis uses document-ref hash) |
| 8 | checkpoint-hash | Hash binding all fields together |
| 9 | process-proof | SWF proof: algorithm, parameters, input, merkle root, sampled proofs, claimed duration |
| 10 | jitter-binding | Keystroke timing entropy commitment, source descriptors, statistical summary, binding MAC |
| 11 | physical-state | Thermal readings, entropy delta, kernel commitment, inertial samples |
| 12 | entangled-mac | HMAC binding jitter, content, and physical state together |
| 13 | receipts | Tool or self-attestation receipts |
| 14 | active-probes | Galton Board, Reflex Gate, or Spatial Target probe results |
| 15 | hat-proof | Roughtime bracketing proof (time-before and time-after with signatures) |
| 16 | beacon-anchor | External time beacon binding (source URL, round, value) |
| 17 | verifier-nonce | 32-byte freshness nonce from WritersProof CA |
| 18 | lamport-signature | 8,192-byte post-quantum one-shot signature |
| 19 | lamport-pubkey-fingerprint | 8-byte public key fingerprint |
| 20 | posme-proof | Proof of Sequential Memory-bound Effort |

**Checkpoint hash formula:**
```
H("PoP-Checkpoint-v1" || prev_hash || content_hash || CBOR(edit_delta)
  || [CBOR(jitter_binding)] || [CBOR(physical_state)]
  || process_proof.merkle_root)
```

#### Jitter Binding (7-Key Structure)

The jitter binding captures the physical entropy of human typing:

| Key | Field | Content |
|-----|-------|---------|
| 1 | entropy-commitment | Hash, timestamp, and link to previous commitment |
| 2 | source-descriptors | Input device type, weight, fingerprint, transport calibration |
| 3 | jitter-summary | Sample count, mean/std/CV of intervals, percentiles, entropy bits, Hurst exponent |
| 4 | binding-mac | HMAC-SHA256 tying jitter to document hash, keystroke count, and timestamp |
| 5 | raw-intervals | Microsecond-resolution keystroke intervals (optional, compressed) |
| 6 | active-probes | Galton Board absorption test and Reflex Gate backspace-latency test |
| 7 | labyrinth-structure | Takens delay-coordinate attractor: embedding dimension, time delay, attractor points, Betti numbers, Lyapunov exponent, correlation dimension |

**Biological invariants validated:**
- Hurst exponent: human range 0.55-0.85 (white noise = 0.5, scripted = 1.0)
- 1/f spectral slope: alpha in [0.8, 1.2]
- Coefficient of variation: 0.15-0.6 (robotic < 0.15)
- Reflex Gate: backspace-after-typo latency in 150-400ms range
- Galton Board: binomial absorption coefficient
- Delay-coordinate attractor: Lyapunov exponent and correlation dimension consistent with chaotic (non-deterministic) dynamics

#### Biology Invariant Claim

A structured sub-object quantifying biological plausibility:

- **Validation status**: empirical, theoretical, or unsupported
- **Millibits**: 0-10,000 (1,000 = 1 bit of discriminating information between human and non-human)
- **Scoring weights**: Hurst exponent, pink noise, error topology, cadence (weighted sum, clamped 0-10,000)
- **Anomaly flags**: WhiteNoiseHurst, PredictableHurst, RoboticCadence, SpectralAnomaly, ErrorTopologyFail, InsufficientData, TemporalDiscontinuity, VelocityAnomaly

### .cwar — Attestation Result

The attestation result format represents a verifier's assessment of an evidence packet. CBOR tag `1129791826` (ASCII "CWAR"). MIME type: `application/vnd.writersproof.cwar+cbor`. File extension: `.cwar`.

| CBOR Key | Field | Description |
|----------|-------|-------------|
| 1 | version | Must be 1 |
| 2 | evidence-ref | SHA-256 of the `.cpoe` packet being assessed |
| 3 | verdict | 1=Authentic, 2=Inconclusive, 3=Suspicious, 4=Invalid |
| 4 | attestation-tier | T1-T4 hardware assurance level |
| 5 | chain-length | Number of checkpoints (must be non-zero) |
| 6 | chain-duration | Total session duration in seconds |
| 7 | entropy-report | Timing, revision, and pause entropy with threshold checks (timing >= 3.0, revision >= 3.0, pause >= 2.0 bits) |
| 8 | forgery-cost-estimate | Estimated adversarial cost across SWF, entropy, hardware, and total (in USD or CPU-hours) |
| 9 | absence-claims | Assertions about what did NOT happen during the session (ComputationallyBound, MonitoringDependent, Environmental) |
| 10 | warnings | Array of warning strings |
| 11 | verifier-signature | COSE_Sign1 over the result |
| 12 | created | Epoch milliseconds |
| 13 | forensic-summary | Flags triggered vs. evaluated, affected vs. total checkpoints, per-flag detail |
| 14 | confidence-tier | PopulationReference, Emerging, Established, or Mature |
| 15 | effort-attribution | Human fraction, human checkpoints, receipt checkpoints, tool-attributed characters vs. total |

### WAR — Written Authorship Report

An ASCII-armored block (`-----BEGIN CPoE WAR-----`) containing a multi-stage hash seal:

- **H1**: `SHA-256(document || checkpoint_root || declaration)`
- **H2**: `SHA-256(H1 || jitter || public_key)`
- **H3**: `SHA-256(H2 || vdf_output || document)`
- **H4**: Ed25519 signature of H3

WAR versions:
- **WAR/1.0** — basic seal
- **WAR/1.1** — entangled mode: each checkpoint's VDF input = `hash(previous_vdf_output || jitter_hash || content_hash || ordinal)`, making the chain non-parallelizable
- **WAR/2.0** — EAR (Entity Attestation Result) appraisal format per draft-ietf-rats-ear

### Process Declaration

When full behavioral evidence is unavailable, the author can produce a signed declaration documenting:

- **Input modalities** — keyboard, dictation, paste, etc., each with a usage percentage
- **AI tools** — tool name, purpose, extent (none / minimal / moderate / substantial), and which sections were affected
- **Collaborators** — other contributors
- **Legal statement** — free-form declaration of authorship

The declaration is signed with Ed25519 over `SHA-256("cpoe-declaration-v1" || canonical_payload)`. A declaration alone produces a Declared-tier attestation; it establishes identity commitment and timestamp but does not constitute forensic proof.

## Security Architecture

### Key Hierarchy

The system uses a four-tier key derivation hierarchy with forward and backward secrecy:

**Tier 0 — Identity Root.** Derived from the device's Physical Unclonable Function (Secure Enclave on macOS, TPM on Windows/Linux) via `HKDF-SHA256(puf_response, "cpoe-identity-v1", "master-seed")`. The master key is never stored; it is re-derived on demand from hardware. The public key and an 8-byte fingerprint form the `MasterIdentity`.

**Tier 1 — Session Key.** Per-session Ed25519 key derived from master key + random 32-byte session ID + timestamp via `HKDF-Expand(master_key, "cpoe-session-v1", session_id || timestamp)`. The master key signs a `SessionCertificate` binding the session key to a specific document hash. Destroyed at session end.

**Tier 2 — Ratcheting Checkpoint Key.** Initialized as `HKDF-Expand(session_key, "cpoe-ratchet-init-v1", "")`, then advanced at each checkpoint: `HKDF-Expand(current_ratchet, "cpoe-ratchet-advance-v1", checkpoint_hash)`. Per-checkpoint signing key = `HKDF-Expand(ratchet_current, "cpoe-signing-key-v1", "")`. Compromising one checkpoint key does not reveal past or future keys.

**Behavioral Key Layer.** A rolling entropy pool fed by per-keystroke SHA-256 timestamp digests. When the user is idle for >30 seconds, HKDF re-derives the active key from master key + entropy pool using `"cpoe-behavioral-entropy-v1"`, ensuring that behavioral patterns are continuously folded into the cryptographic state.

### Cryptographic Primitives

| Purpose | Primitive | Notes |
|---------|-----------|-------|
| Checkpoint/packet signing | Ed25519 (ed25519-dalek) | 64-byte signatures |
| Post-quantum hedge | Lamport one-shot signatures | 8,192-byte sig per checkpoint; constant-time verification via `subtle::ConstantTimeEq` |
| Hardware signing | ECDSA-P256 (COSE/Secure Enclave) | Hardware-bound, non-exportable |
| Event integrity | HMAC-SHA256 | Per-event MAC with session HMAC key |
| Hash chains | SHA-256 | Checkpoint chain, event chain, jitter chain |
| Fast hashing | BLAKE3 | Internal operations |
| Key derivation | HKDF-SHA256 | All key derivation uses domain-separated HKDF |
| IPC encryption | ChaCha20-Poly1305 | Encrypted channels between processes |
| Offline queue encryption | ChaChaPoly | At-rest encryption of sync queue |
| Key material protection | Zeroize crate | All key material zeroed on drop |
| Recovery phrase | BIP-39 mnemonic | Human-readable identity backup |
| Sequential work proof | Iterated SHA-256 / Argon2id / PoSME | Non-parallelizable time proofs |

### Hardware Security Integration

**Secure Enclave (macOS):** Keys are created inside the Secure Enclave and never leave it. Seal operations use v5 format: a 32-byte seal nonce is signed by the SE key, the SHA-256 of the signature becomes the AES key material, and data is encrypted with ChaCha20-Poly1305 using a random 12-byte AEAD nonce. Legacy v4 format (XOR cipher) is explicitly rejected.

**TPM 2.0 (Windows/Linux):** Provides quote, bind, sign, verify, seal, and unseal operations. Monotonic counters are folded into checkpoint ratchet derivation so hardware-counter skips are detectable. Reset count tracking detects mid-session reboots.

**Platform detection priority:** Secure Enclave > Windows TPM 2.0 > Linux TPM 2.0 > Software fallback. The UI clearly distinguishes hardware-backed from software-only key storage.

### Event Integrity

Every `SecureEvent` stored in SQLite has two integrity fields:

- **Event hash:** `SHA-256("cpoe-event-v2" || device_id || timestamp || path || content_hash || file_size || size_delta || previous_hash)`. Each event commits its predecessor's hash, forming a cryptographic chain where deletion or modification of any event invalidates the chain.
- **HMAC:** `HMAC-SHA256(hmac_key, same_fields)`. The HMAC key is derived as `SHA-256("cpoe-hmac-key-v1" || seed)` and stored securely.

An `integrity` table holds a separate HMAC over the chain metadata itself. Incremental verification on store open re-checks only events added since the last verification pass, using constant-time comparison.

### Verifiable Delay Functions (VDF)

VDF proofs demonstrate that real wall-clock time elapsed between checkpoints. The system supports multiple algorithms:

- **Simple VDF:** Iterated SHA-256 chain. Output = `SHA-256^iterations(input)`. Verification requires recomputing the entire chain; there is no shortcut.
- **RFC-compliant VDF:** Wesolowski proof with calibration attestation. Duration bounds enforced: claimed duration must be in [0.5x, 3.0x] of expected (from calibrated iterations-per-second). Calibration freshness window: 24 hours.
- **Argon2id SWF:** Memory-hard sequential work function using Argon2id with Merkle root commitments.
- **PoSME:** Proof of Sequential Memory-bound Effort, with configurable reads-per-step, challenges, and recursion depth.

In WAR/1.1 entangled mode, each checkpoint's VDF input includes the previous VDF output, the jitter hash, the content hash, and the ordinal. This means the VDF chain cannot be parallelized; each step depends on the previous step's output AND the actual behavioral evidence, binding time to process.

### Merkle Mountain Range (Anti-Deletion)

All checkpoints are indexed in a Merkle Mountain Range. The MMR root is committed into each checkpoint's hash preimage via domain separator `"cpoe-mmr-root-v1"`. Deleting any checkpoint from the middle of the chain changes the MMR root, which invalidates all subsequent checkpoint hashes. This prevents selective evidence deletion while keeping verification efficient (log-time inclusion proofs).

### Anti-Analysis Measures

- `ptrace(PT_DENY_ATTACH)` on macOS prevents debugger attachment to the sentinel process.
- `is_debugger_present()` checks on macOS (kinfo_proc P_TRACED flag) and Windows (IsDebuggerPresent). Debugger detection downgrades TrustTier to Local and adds a limitation marker.
- `mlock()` on key material prevents the OS from swapping secrets to disk.
- All key types implement `ZeroizeOnDrop`.
- Atomic file writes use unpredictable temporary filenames (via `tempfile::NamedTempFile`) with fsync + rename, preventing symlink attacks. Owner-only permissions are applied on both Unix and Windows.

## Anti-Forgery Measures

### Forgery Cost Model

The system computes an explicit economic estimate of the cost to forge evidence, broken down by component:

| Component | Cost Basis | Notes |
|-----------|-----------|-------|
| VDF chain | Sequential wall-clock seconds | Cannot be parallelized |
| Checkpoint chain | Elapsed session time | Must simulate real-time editing |
| Jitter entropy | 0.1 CPU-seconds per sample x count | Human dynamics are statistically expensive to fake |
| Hardware attestation | Infinite | Requires physical device possession |
| Behavioral fingerprint | 3,600 CPU-seconds | Hurst, 1/f, cadence profiles |
| Cross-modal consistency | O(n^2) pairwise x 60 CPU-seconds | All evidence channels must agree |
| External time anchors | Infinite | RFC3161/Roughtime cannot be backdated |
| Content-key entanglement | 2x chain duration | Content changes are bound to keys |

Difficulty tiers: Trivial (<60s) / Low (<1h) / Moderate (<24h) / High (>24h) / Very High (hardware-bound or infinite). The weakest-link component is explicitly identified.

### Cross-Modal Consistency Checks

Five independent consistency checks across evidence channels that an attacker would need to simultaneously satisfy:

1. **Content growth rate** — gross character additions / session duration must be <= 15 chars/sec. Zero additions with non-empty document = definitive failure (catches AI-pasted documents opened for "witnessing").
2. **Edit-checkpoint ratio** — edit events per checkpoint in [0.5, 200]; out of range suggests fabricated checkpoints.
3. **Jitter-edit coherence** — ratio of edit events to jitter samples >= 0.02; very low means injected jitter from a different session.
4. **Temporal span alignment** — jitter timestamps must bracket edit timestamps within 120 seconds.
5. **Jitter-content entanglement** — keystroke count / document length >= 0.5, jitter count / keystrokes >= 0.3.

Verdict: Consistent (0 failures), Marginal (1-2), Inconsistent (>=3).

### Tampering Detection

Eight concurrent detectors, each represented as a bit flag:

| Flag | Detection | Threshold |
|------|-----------|-----------|
| IMPOSSIBLE_SPEED | Mean IKI < 40ms | >300 WPM sustained over >=10 events |
| MONOTONIC_TIMING | IKI CV < 10% | Scripted or pasted content |
| SLEEP_RECOVERY | Post-sleep IKI within 5% of pre-sleep | Unrealistic session resume |
| ORPHANED_KEYSTROKES | >5 keystrokes before focus | Events without application context |
| CHECKSUM_MISMATCH | Hash chain break | Deferred to checkpoint layer |
| NONCE_REPLAY | Duplicate nonce | Deferred to NonceManager |
| TIMESTAMP_VIOLATION | Non-monotonic or >5s future | Clock manipulation |
| SIGNATURE_FAILURE | Invalid Ed25519 | Deferred to crypto layer |

### Entropy Validation

Shannon entropy of IKI distributions is continuously monitored:

- Minimum threshold: 1.5 bits/keystroke
- Assessment levels: High (>=2.5), Medium (>=1.5), Low (CV < 15% or mean IKI outside 50-2000ms), Critical (<1.5)
- Pattern detectors: monotonic timing (std_dev < 5% of mean), zero-variance windows (<5ms std dev in 500ms windows), rapid bursts (mean IKI < 200ms), discontinuous gaps

### Keystroke Validation Gate

Every keystroke passes through `validate_keystroke_event()`. Confidence < 0.1 causes the keystroke to be uncounted and removed from jitter samples. An adversary injecting synthetic keystrokes through OS APIs cannot accumulate valid evidence.

### Steganographic Mouse Watermark

A covert verification channel: HMAC-SHA256 keyed by the session signing key is embedded into mouse event timing as sub-perceptual delays (500-2000 microseconds, below the ~10ms human perception threshold). Each event's jitter depends on the previous event's hash, creating an independently verifiable chain. Verification replays the chain and checks values within 100-microsecond tolerance.

### Hardware Co-Signature Chain

Each checkpoint can carry a hardware co-signature where the entangled hash includes the previous hardware signature:
```
SHA-256("cpoe-hw-cosign-v1" || doc_hash || sw_binding || tpm_clock
  || monotonic_counter || device_id || public_key || prev_hw_signature)
```
This creates a hardware-signed causal chain that an attacker cannot forge without continuous access to the physical TPM/Secure Enclave.

## Privacy

WritersLogic is designed around a fundamental principle: prove the process without exposing the product. The system never needs to see, store, or transmit the actual content being written.

### What Is Captured

- **Keystroke timing** — inter-key intervals in microseconds, keyboard zone, dwell time, flight time. The key identity (which character was pressed) is NOT recorded.
- **Document hashes** — SHA-256 of document content at each checkpoint. The hash proves the document existed in a specific state; it cannot be reversed to recover the content.
- **Edit deltas** — characters added, characters deleted, operation count. Aggregate statistics, not the actual text.
- **Application metadata** — bundle identifier and document path of the focused application. Window titles are obfuscated for applications without file path access.
- **Mouse movement statistics** — aggregate micro-movement data during typing, not cursor positions or click targets.

### What Is NOT Captured

- **Keystrokes themselves** — the system records when keys are pressed, not which keys. There is no keylogger.
- **Screen content** — no screenshots, no screen recording, no OCR.
- **Document text** — the system hashes the document; it does not read or store the content. Document embedding in evidence packets is optional and explicit.
- **Clipboard content** — paste events are detected and sized, but clipboard text is not read or stored.
- **Network activity** — the system does not monitor browsing, email, or other network traffic.
- **Audio or video** — no microphone or camera access.

### Data Sovereignty

- All evidence is stored locally in an encrypted SQLite database on the user's device.
- Cloud sync (to Supabase) is opt-in, requires a Pro subscription, and syncs only evidence metadata and fingerprint statistics, not document content.
- The offline sync queue is encrypted with ChaChaPoly, limited to 50 operations, and entries expire after 7 days.
- Evidence export is always initiated by the user and requires biometric authentication.

### Privacy-Preserving Verification

When a third party verifies a WritersProof attestation (e.g., by scanning a proof block or visiting the verification URL), the verification is privacy-preserving:

1. The verifier's browser normalizes the text and computes SHA-256 **client-side** (via WASM).
2. Only the hash is sent to the WritersProof API.
3. The server returns the attestation tier, confidence, timestamp, and evidence metadata.
4. **The text itself never leaves the browser.**

Text normalization before hashing strips whitespace, punctuation, symbols, and emoji; retains Unicode letters and ASCII digits; lowercases everything. This makes verification resilient to formatting differences from copy/paste.

### Voice Fingerprint Privacy

The opt-in voice fingerprint feature stores only statistical typing patterns (WPM, IKI distribution, think time, correction rate). Raw text is never captured. All fingerprint data stays on-device unless the user explicitly enables cloud sync. The fingerprint can be exported as JSON and reset at any time.

### Shadow Buffers

For applications that do not provide file path access, the sentinel creates encrypted in-memory shadow buffers with random hex identifiers and obfuscated window titles. This prevents document identification through metadata even in the sentinel's own internal state.

### Debugger and Integrity Protections

Anti-analysis measures (ptrace denial, debugger detection) protect user evidence from extraction by malicious software. Memory-locked key material cannot be swapped to disk. Atomic writes with restrictive permissions prevent other processes from reading evidence files during creation.

## Standards Alignment

WritersLogic is designed to interoperate with and complement existing content provenance, identity, and attestation standards.

### IETF RATS (Remote ATtestation ProcedureS)

The core protocol implements **draft-condrey-rats-pop** (Proof-of-Process), built within the IETF RATS architecture:

- **EAT (Entity Attestation Token)** — draft-ietf-rats-eat: Evidence packets use the EAT profile mechanism. Profile URI: `urn:ietf:params:rats:eat:profile:pop:1.0`.
- **EAR (Entity Attestation Result)** — draft-ietf-rats-ear: WAR/2.0 attestation results follow the EAR appraisal format with AR4SI status codes (None, Affirming, Warning, Contraindicated) and trust vector components.
- **AR4SI (Attestation Results for Secure Interactions)** — draft-ietf-rats-ar4si: Trust vectors cover instance identity, configuration, executables, file system, hardware, runtime opaque, storage opaque, and sourced data. Private-use EAR keys (70001-70011) carry PoP-specific extensions.

### CBOR / COSE (RFC 8949 / RFC 9052)

All wire formats use CBOR with integer-keyed maps for compact encoding. Signatures use COSE_Sign1 (single-signer). Custom CBOR tags are registered in the vendor space. Evidence packets and attestation results are fully self-describing CBOR documents that can be parsed by any CBOR implementation.

### C2PA (Coalition for Content Provenance and Authenticity)

WritersLogic is positioned as an **evidence source for C2PA**, not a replacement:

- C2PA defines a content provenance manifest format (ISO 19566-5) that travels with media files, primarily targeting images and video.
- WritersLogic produces behavioral authorship evidence that can be referenced from within a C2PA manifest as an external assertion.
- A C2PA manifest answers "what transformations were applied to this content?" WritersLogic answers "was a human observed creating this content?" These are complementary claims.
- **Current gaps relative to C2PA:** no JUMBF container format, no X.509 certificate chain, no RFC 3161 timestamp server integration. These are architectural differences (WritersLogic uses CBOR/COSE and Ed25519 rather than JUMBF and X.509), not missing features.

### W3C Standards

- **DID Core 1.0 (Decentralized Identifiers):** The `author_did` field (CBOR key 20) supports `did:` URIs, enabling decentralized identity binding. The `did:webvh` method is implemented via feature flag.
- **Verifiable Credentials Data Model 2.0:** Evidence packets can be referenced from W3C Verifiable Credentials. The `credential_reference` field carries a VC reference. COSE-based VC securing is implemented per W3C VC COSE specification.
- **W3C AI Disclosure:** Standards mapping for AI tool usage transparency is implemented, aligning with W3C guidance on AI-generated content disclosure.

### OpenID for Verifiable Credentials (OID4VCI)

Implemented for credential issuance workflows, enabling WritersProof attestations to be issued as verifiable credentials through standard OpenID Connect flows.

### Creator Ontology and Content Credentials

- **IPTC (International Press Telecommunications Council):** Metadata mappings for news and media content provenance.
- **JPEG Trust (ISO 21617):** Integration for image-based content provenance.
- **CAWG (Creator Assertions Working Group) Identity Assertion v1.2:** Partial implementation. Current gap: no `identity_assertion` JUMBF box (architectural difference due to CBOR/COSE vs. JUMBF format choice).
- **ORCID:** Implemented for academic and research author identification.
- **DIF Presentation Exchange:** Implemented for credential presentation and verification workflows.

### Regulatory Compliance

- **EU AI Act Article 50** (effective August 2026): AI-generated content labeling requirements are implemented through mandatory tool declarations and AI scoring.
- **NIST AI RMF 1.0 (AI Risk Management Framework):** Risk mappings implemented in the standards compliance module.
- **NIST AI 100-4 (Reducing Risks Posed by Synthetic Content):** Aligned through behavioral detection and process attestation.
- **ISO/IEC 42001 (AI Management Systems):** Compliance mappings implemented.

### Industry Standards

- **WGA MBA / SAG-AFTRA:** Mapped for entertainment industry AI disclosure requirements.
- **ToIP EGF / TRQP (Trust over IP Governance Framework):** Planned.
- **IETF SCITT (Supply Chain Integrity, Transparency, and Trust):** Partial implementation for transparency log integration.

### Time Attestation

- **Roughtime (draft-ietf-ntp-roughtime):** Bracketing proofs with time-before and time-after signatures from Roughtime servers, providing cryptographic time bounds without trusting a single clock.
- **External Beacons:** drand and NIST randomness beacons can be anchored into checkpoints, providing publicly verifiable timestamps.

## Browser Extension

The WritersProof browser extension extends authorship witnessing to web-based writing platforms.

### Supported Platforms

Default-enabled: Google Docs, Overleaf, Medium, Notion, Craft, Coda, ClickUp, Nuclino, StackEdit, HackMD, Hemingway Editor, Riseup Pad, Write.as, WordPress, Ghost, Substack. Custom domains can be added with wildcard support.

### Two Operating Modes

**Native Mode** — when the desktop application is running, the extension connects via Chrome Native Messaging (`com.writerslogic.witnessd`). Full capabilities: hardware attestation, VDF proofs, Secure Enclave, encrypted channels, Ed25519 signatures. Badge: green.

**Standalone Mode** — fallback when the desktop application is not available. SHA-256 hash chain for checkpoint linking, HMAC-SHA256 session sealing. Reduced evidence strength. Badge: orange with "S" indicator.

### Behavioral Capture

- Pre-allocated `Float64Array(51)` ring buffer for keystroke timestamps with keyboard zone assignment across five zones.
- `MutationObserver` with adaptive debounce (2,000ms normal / 3,000ms for large documents) and SHA-256 change detection.
- SPA navigation support via `history.pushState` / `replaceState` override.
- Minimum change threshold: 5 characters. Maximum document size: 10MB.

### AI Tool Copy Detection

The extension monitors clipboard copy events from known AI tool hosts: `chat.openai.com`, `chatgpt.com`, `claude.ai`, `gemini.google.com`, `copilot.microsoft.com`. Copy events from these domains are recorded as `ai_content_copied` signals in the evidence chain.

### Content Tiers

- **Core** — no timing jitter
- **Enhanced** (default) — standard behavioral capture
- **Maximum** — adds mutation rate tracking

### Export

Standalone mode supports JSON export and HTML report generation with checkpoint tables, hash chain integrity status, and trust level explanations.
